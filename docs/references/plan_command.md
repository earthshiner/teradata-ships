# `ships plan` — detect-and-recommend

Issue [#379](https://github.com/earthshiner/teradata-ships/issues/379).

`ships plan` inspects a raw source DDL tree, auto-answers the questions it can
read straight off the files, and emits the recommended `ships` command sequence,
a per-step rationale, and a machine-readable `plan.json` — without an
interactive prompt. It is the non-interactive sibling of the
[SHIPS Navigator](../../tools/navigator/README.md) and the CLI wizard
([#381](https://github.com/earthshiner/teradata-ships/issues/381)); all three
share the declarative model in
[`decision-tree.yaml`](../../tools/navigator/decision-tree.yaml) and the same
plan-emission engine (`td_release_packager.packaging_plan`).

## Usage

```bash
ships plan --source /raw/omr-ddl --project /proj/omr --env DEV,TST \
    --name create_objects --json plan.json
```

| Flag | Meaning |
|------|---------|
| `--source` | Raw source DDL directory to inspect (required) |
| `--project` | Target SHIPS project dir (default: the source dir) |
| `--env` | Target environments, e.g. `DEV,TST,PRD` |
| `--name` | Package name (default: `create_objects`) |
| `--mode` | `quick` (one `ships process` per env) or `detailed` (each step) |
| `--strict` | Recommend `--strict` on `process` |
| `--scaffolded` | Project already scaffolded — omit the scaffold step |
| `--no-generate` | Skip the view-layer generate step |
| `--json PATH` | Write the machine-readable `plan.json` |

## What it auto-detects

Detection is read-only and conservative — SQL is scanned as text, never run, and
every decision is reported with its evidence:

| Signal | How |
|--------|-----|
| `source.type` / `source.dir` | a filesystem path was given |
| `tokens.already` | any file contains `{{TOKEN}}` placeholders |
| `atomic.eponymous` | every DDL file holds exactly one object (else SHIPS auto-splits) |
| DCL / DML present | `GRANT`/`REVOKE` and `INSERT`/`UPDATE`/`DELETE`/`MERGE` (informational) |

CLI flags override any detected or defaulted answer.

## Output

The plan prints three sections — **Detected** (evidence), **Recommended
commands** (the ordered `ships …` sequence), and **Why each step** (rationale).
With `--json` it also writes `plan.json`, the canonical answers snapshot the
HTML Navigator and CLI wizard accept, so a plan generated by one front end can
be replayed by another.

Config-skeleton *files* (env `.conf`, `tokenise.conf`, `ships.yaml`) are authored
by the `scaffold` / `bootstrap-env-config` commands the plan recommends — the
plan points at them rather than duplicating that authoring logic.

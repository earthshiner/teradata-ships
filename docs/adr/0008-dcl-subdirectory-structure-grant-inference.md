# ADR 0008: DCL Subdirectory Structure and Intent-Based Grant Inference

## Status

Accepted | 2026-04-02

## Context

Teradata access control for a multi-database data product requires
several categories of `GRANT` statement:

1. **Role grants.** `GRANT {privilege} ON {database} TO {role}`.
   Grants a named role access to a database. Roles aggregate
   privileges; users are then granted roles rather than direct
   privileges.

2. **User grants.** `GRANT {role} TO {user}`. Assigns a role to
   a user or service account. May also include direct database
   grants to functional accounts (ETL users, reporting users).

3. **Inter-database grants.** `GRANT SELECT ON {source_db} TO
   {target_db}`. In Teradata, a view database requires `SELECT
   WITH GRANT OPTION` on its backing table database to allow the
   view to be queried by downstream consumers. These grants are
   the structural connective tissue of a multi-database data
   product.

In early SHIPS prototypes, all `GRANT` statements were placed in
a flat `DCL/` directory alongside the DDL files. This created
several problems:

- A reviewer inspecting the DCL directory could not determine
  at a glance whether a file contained a role grant, a user
  grant, or an inter-database grant. All three require different
  review contexts: role grants affect access for all users of
  that role; inter-database grants affect query capabilities
  across database boundaries.
- Wave 2 ordering within DCL was underdetermined. Roles must
  exist before they are granted to users. Users must exist before
  they receive role assignments. Inter-database grants must come
  after both sides of the grant exist. A flat directory provides
  no mechanism for the deployer to derive this ordering.
- Cross-database grants were being written manually, which
  produced inconsistencies: some developers granted `SELECT`,
  some granted `SELECT, INSERT` (unnecessary for view databases),
  some forgot `WITH GRANT OPTION` (required for view databases
  to propagate SELECT to their consumers).

The DCL structure needed to:

- Separate the three grant categories into distinct, reviewable
  groups.
- Provide the deployer with enough structural information to
  derive intra-Wave 2 ordering without parsing grant content.
- Standardise inter-database grants so they are generated
  consistently rather than written by hand.

A separate question was whether the `DBC` database should be
tokenised. `DBC` is the canonical system catalog database in
Teradata, but some sites proxy it through a filtered views
database to control exposure. If `DBC` is hard-coded in grant
statements, the package is not portable to sites that have
renamed or proxied the catalog.

## Decision

### DCL Subdirectory Structure

The `DCL/` subdirectory within each database's payload directory
is organised into three mandatory subdirectories:

```
payload/{database}/DCL/
    roles/          # GRANT {privilege} ON {database} TO {role}
    users/          # GRANT {role} TO {user}; direct user grants
    inter_db/       # GRANT {privilege} ON {source_db} TO {target_db}
```

Each subdirectory contains `.dcl` files following the atomic
eponymous convention (ADR 0004 adapted for DCL):
`{source}.{target}.dcl` for inter-database grants;
`{database}.{role}.dcl` for role grants; `{database}.{user}.dcl`
for user grants.

The deployer executes Wave 2 in the order: `roles/` first,
`users/` second, `inter_db/` third. This ordering is structural —
derived from subdirectory name, not from file content.

### Intent-Based Cross-Database Grant Inference

Inter-database grants in `DCL/inter_db/` are generated by
`infer_grants.py` using the following rules:

1. **SQL verb determines privilege.** The inference engine
   inspects the DDL object types in each database to determine
   the access pattern:

   | Database content | Inferred grant |
   |-----------------|---------------|
   | Tables (`.tbl`) | `GRANT SELECT ON {db} TO {consumer}` |
   | Views (`.viw`)  | `GRANT SELECT ON {db} TO {consumer}` |
   | Procedures (`.spl`) | `GRANT EXECUTE PROCEDURE ON {db} TO {consumer}` |
   | Macros (`.mcr`) | `GRANT EXECUTE MACRO ON {db} TO {consumer}` |

2. **WITH GRANT OPTION always.** All inter-database grants use
   `WITH GRANT OPTION`. This allows the granted privilege to
   be re-granted by the receiving database — necessary for view
   databases to propagate `SELECT` to their downstream
   consumers. Without `WITH GRANT OPTION`, a view in
   `{DOM_DATABASE_V}` that references `{DOM_DATABASE_T}` cannot
   be queried by a user who has only `SELECT ON {DOM_DATABASE_V}`.

3. **Container-level only.** Grants are at the database level,
   never at the table or column level. Table- and column-level
   grants are a governance concern managed outside SHIPS.

4. **DBC is always tokenised.** The system catalog database is
   referenced as `{{DBC_DATABASE}}` in source grant files, mapping
   to `DBC` by default. Sites that proxy DBC through a filtered
   views database can override this token without touching any
   grant file.

5. **Validation.** `validate_grants.py` checks every
   inter-database grant file for:
   - Correct `WITH GRANT OPTION` suffix.
   - Container-level (not object-level) grant scope.
   - No grants from a table database to another table database
     (a red flag: business logic should flow through view
     databases, not between raw table stores).
   - Presence of the `{{DBC_DATABASE}}` token wherever `DBC`
     would otherwise appear.

### Inference Engine Location

`infer_grants.py` is invoked as part of the Inspect phase
(`ships inspect --fix-grants`). It writes inferred grant files
to `payload/{database}/DCL/inter_db/`. Inferred files are
stamped with a comment header identifying them as generated
rather than manually authored. Manual override is permitted:
a hand-authored `.dcl` file in `inter_db/` takes precedence
over the inferred file for the same source-target pair.

## Consequences

**Positive**

- Intra-Wave 2 ordering is structurally encoded. The deployer
  does not parse grant content to determine sequencing; it reads
  the subdirectory.
- Consistent inter-database grants. `WITH GRANT OPTION` is
  never accidentally omitted. The privilege tier (`SELECT` vs
  `EXECUTE PROCEDURE`) is derived from what the database
  actually contains, not from developer recall.
- The `DBC` tokenisation closes the portability gap for sites
  with non-standard catalog configurations. A SHIPS package is
  portable to any Teradata site by adjusting the token map.
- Code review of DCL is focused: a reviewer looking at a
  Wave 2 failure can go directly to `inter_db/` without
  scanning all grant files.

**Negative**

- Three subdirectories under `DCL/` add directory depth to the
  payload. For simple single-database deployments (no inter-
  database grants), `inter_db/` is always empty. This is minor
  friction.
- The inference engine produces `SELECT` grants for both table
  databases and view databases. A consumer that should only have
  access to the view layer — not the table layer — requires an
  explicit exclusion. The inference engine does not model the
  data product's intended access tier.
- Manually-authored grant files in `inter_db/` that conflict
  with inferred files require the developer to explicitly
  acknowledge the override. The mechanism (file presence takes
  precedence) is simple but not enforced via a conflict warning
  in the current implementation.

**Neutral**

- The `roles/` and `users/` subdirectories contain manually
  authored files only. The inference engine does not generate
  role or user grants. These are environment-specific and must
  be provided by the project.
- The `.dcl` extension for grant files was chosen to distinguish
  them from DDL object files (`.tbl`, `.viw`, etc.) and to
  signal that these are access control statements. The extension
  has no Teradata semantic meaning.

## Alternatives considered

**Flat DCL directory with content-based ordering.** Rejected:
requires parsing grant content to derive execution order.
Parsing `GRANT` statements to determine whether a file is a
role grant, user grant, or inter-database grant is feasible but
adds complexity and creates a parsing dependency. Structural
ordering via subdirectory names is simpler and more reliable.

**Separate DCL directory per database under the top-level
payload.** Considered: a `payload/DCL/` directory at the top
level rather than under each database. Rejected: co-locating
grants with the database they govern (under `payload/{database}/
DCL/`) makes the relationship between grants and their target
database explicit in the directory structure. A top-level DCL
directory mixes grants for all databases into a single flat
namespace.

**Privilege matrix configuration in `ships.yaml`.** Considered:
a declarative privilege matrix specifying which database should
have what access to which other database. Rejected: the matrix
would duplicate information already expressed in the data
product's DDL structure. Intent-based inference derives the
same result without a separate configuration artefact.

**Object-level grants.** Rejected at project inception: object-
level grants (table-level, column-level) are a governance and
security engineering concern that requires input from the data
product owner and the security team. SHIPS produces the structural
inter-database grants; object-level grants are out of scope.

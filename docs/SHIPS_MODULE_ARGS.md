# SHIPS Module and Argument Reference

This document describes the command-line surfaces and major Python modules in `teradata-ships`. It is intended for developers, operators, and autonomous agents that need to understand which component owns each part of the SHIPS workflow.

## Entry-point argument reference

## `td_release_packager` / `ships` packaging CLI

SHIPS — Scaffold, Harvest, Inspect, Package, Ship. Standardised Teradata DDL deployment methodology.

| `-v, --verbose` | no | `False` | `` |  |

### `harvest`

```text
usage: td_release_packager harvest [-h] [--source SOURCE] --project PROJECT
                                   [--token-map TOKEN_MAP]
                                   [--generate-token-map]
                                   [--env-prefix ENV_PREFIX]
                                   [--apply-tokens APPLY_TOKENS]
                                   [--no-detect-tokens] [--force]
                                   [--keep-existing] [--auto-tokenise]
                                   [--reconcile] [--json-out JSON_OUT]
```

| Argument | Required | Default | Choices | Description |
|---|---:|---|---|---|
| `--source` | no | `` | `` | Directory containing raw DDL files. Required for normal harvest; ignored in --reconcile mode. |
| `--project` | yes | `` | `` | Target project directory (must be scaffolded). |
| `--token-map` | no | `` | `` | Path to token_map.conf — applies literal → {{TOKEN}} substitutions during harvest. Generate one with --generate-token-map first, review it, then pass it here. |
| `--generate-token-map` | no | `False` | `` | Scan for hardcoded database names and write a token_map.conf to the project's config/ directory. Requires --env-prefix to derive token names. |
| `--env-prefix` | no | `` | `` | Optional environment prefix to strip when deriving token names (e.g. 'A_D01'). Used with --generate-token-map to turn 'A_D01_OMR_STD' into '{{OMR_STD}}'. If omitted, the full database name becomes the token (e.g. 'CORE_STD' → '{{CORE_STD}}'). |
| `--apply-tokens` | no | `` | `` | (Legacy) Comma-separated name=token pairs. Prefer --token-map instead. E.g. 'DEV01_STD={{STD_DATABASE}},DEV01_SEM={{SEM_DATABASE}}' |
| `--no-detect-tokens` | no | `False` | `` | Skip hardcoded name detection. |
| `--force` | no | `False` | `` | Overwrite existing files in the payload. Redundant under the default clean-payload mode (nothing pre-exists to overwrite); only meaningful alongside --keep-existing, where it governs per-file collisions during overlay re-harvest. |
| `--keep-existing` | no | `False` | `` | Skip the pre-harvest payload clean and overlay new files on top of whatever is already in payload/database/. The default behaviour wipes harvest-owned files first (preserving .gitkeep and control files starting with '_' like a user-curated _order.txt) so the payload always reflects current source state without orphaned artefacts. |
| `--auto-tokenise` | no | `False` | `` | Auto-detect hardcoded database names and apply token substitutions in a single pass — no manual token_map.conf review step required. The token map is derived automatically from detected candidates (optionally stripped with --env-prefix) and applied immediately. Use in developer mode when speed matters more than reviewing every token. |
| `--reconcile` | no | `False` | `` | Run interactive reconciliation: detect literal/tokenised twin file pairs in the harvested DDL tree and prompt to resolve each. Skips the normal harvest pipeline. Requires --project and config/token_map.conf; --source is ignored. |
| `--json-out` | no | `` | `` | Override the default JSON audit destination (<project>/logs/reconcile_<timestamp>.json) for --reconcile mode. Relative paths resolve under --project. |

### `import-legacy`

Bootstrap a SHIPS project from legacy substitutions. Two mutually exclusive input modes: --script consumes an existing sed substitution script (s/$VAR/value/g rules); --scan-source walks a source DDL tree and auto-discovers $VAR / ${VAR} / &&VAR&& placeholders. Both modes emit a .conf file (token values) and a sed migration script (legacy markers → {{TOKEN}}). --scan-source additionally writes scan_report.md, an audit detail of every discovered token.

```text
usage: td_release_packager import-legacy [-h] (--script SED_FILE |
                                         --scan-source SOURCE_DIR)
                                         [--project PROJECT] --env ENV
                                         [--output-dir OUTPUT_DIR]
```

| Argument | Required | Default | Choices | Description |
|---|---:|---|---|---|
| `--script` | no | `` | `` | Path to a legacy sed substitution script. Use this when your project's pre-SHIPS build harness already has a sed file defining (marker, value) pairs. |
| `--scan-source` | no | `` | `` | Walk a source DDL directory and auto-discover non-SHIPS placeholders ($VAR, ${VAR}, &&VAR&&). Use this when the project has placeholders embedded in source but no sed file to point at -- the .conf values come out empty for you to fill in, and the migration sed converts every discovered marker to its {{TOKEN}} form. NOTE: expects a DIRECTORY (the root of your source DDL), not a single file. |
| `--project` | no | `` | `` | Optional SHIPS project root. When supplied, the discovery resolver consults the project's ships.yaml for any extra extensions to scan. Only meaningful with --scan-source. |
| `--env` | yes | `` | `` | Target environment name (DEV, TST, PRD). |
| `--output-dir` | no | `.` | `` | Output directory (default: current). Files written under <output-dir>/env/<env>.conf and <output-dir>/legacy_migration.sed. In --scan-source mode an additional <output-dir>/scan_report.md is also written. |

### `migrate-source`

Apply a ``legacy_migration.sed`` (generated by ``import-legacy``) to every SQL-bearing file in a source tree, converting legacy substitution markers (``$VAR``, ``${VAR}``, ``&&VAR&&``) to SHIPS ``{{TOKEN}}`` form. Understands only the ``s/LHS/RHS/g`` subset that ``import-legacy`` emits -- not full sed syntax. Run with ``--dry-run`` first to see what would change.

```text
usage: td_release_packager migrate-source [-h] --sed SED_FILE
                                          --source SOURCE_DIR
                                          [--project PROJECT_DIR] [--dry-run]
```

| Argument | Required | Default | Choices | Description |
|---|---:|---|---|---|
| `--sed` | yes | `` | `` | Path to the ``legacy_migration.sed`` produced by ``import-legacy``. Can also be any sed script containing ``s/LHS/RHS/g`` substitution rules. |
| `--source` | yes | `` | `` | Root of the source DDL tree to migrate. Files are updated in place. |
| `--project` | no | `` | `` | Optional SHIPS project root. Consulted by the discovery resolver for ships.yaml extension overrides. |
| `--dry-run` | no | `False` | `` | Preview what would change without writing any files. |

### `decompose-names`

Read a list of literal Teradata database names (from a token_map.conf or a plain names file) and decompose them against the SHIPS grammar {ENV_PREFIX}_{SHIPS_ENV}_{INSTANCE}_{LAYER}_{SECURITY_TIER}_{KIND}. Emits a sectioned .conf file with composition roots and derived names in cascade form, plus a markdown report.

```text
usage: td_release_packager decompose-names [-h] --env ENV
                                           [--output-dir OUTPUT_DIR]
                                           input
```

| Argument | Required | Default | Choices | Description |
|---|---:|---|---|---|
| `input` | yes | `` | `` | Path to a token_map.conf or plain names file (one literal per line). Format auto-detected. |
| `--env` | yes | `` | `` | Target environment name (DEV, TST, PRD). |
| `--output-dir` | no | `.` | `` | Output directory (default: current). Files written under <output-dir>/env/<env>.conf and <output-dir>/decomposition_report.md. |

### `bootstrap-env-config`

Scan an already-tokenised SHIPS project for {{TOKEN}} references and emit a 7-section .conf scaffold with every referenced token parked in section 8 for the user to re-section. Closes the third bootstrap path: when there's nothing to convert (no literals, no legacy script) you just need a starting .conf skeleton.

```text
usage: td_release_packager bootstrap-env-config [-h] --source SOURCE --env ENV
                                                [--output-dir OUTPUT_DIR]
                                                [--force]
```

| Argument | Required | Default | Choices | Description |
|---|---:|---|---|---|
| `--source` | yes | `` | `` | SHIPS project directory (with payload/ already harvested). |
| `--env` | yes | `` | `` | Target environment name (DEV / TST / PRD). |
| `--output-dir` | no | `` | `` | Output directory; .conf written under <output-dir>/env/<env>.conf. Defaults to <source>/config. |
| `--force` | no | `False` | `` | Overwrite an existing .conf file at the target. Without this, the tool refuses to clobber. |

### `process`

Orchestrate the complete SHIPS pipeline in a single command, recording all stage decisions into one run entry in ships.decisions.json.

Developer mode (default): continues past warnings; hard errors are reported but do not abort.
Platform mode (--strict): any stage error immediately aborts the pipeline.

```text
usage: td_release_packager process [-h] --project PROJECT [--source SOURCE]
                                   [--source-github OWNER/REPO]
                                   [--source-ref REF] [--github-token TOKEN]
                                   [--token-map TOKEN_MAP] [--auto-tokenise]
                                   [--env-prefix ENV_PREFIX] [--skip-generate]
                                   [--inspect-config INSPECT_CONFIG]
                                   [--env ENV] [--env-config ENV_CONFIG]
                                   [--name NAME] [--output OUTPUT]
                                   [--format {zip,tar.gz}] [--author AUTHOR]
                                   [--description DESCRIPTION]
                                   [--commit COMMIT] [--strict] [--pause]
```

| Argument | Required | Default | Choices | Description |
|---|---:|---|---|---|
| `--project` | yes | `` | `` | SHIPS project directory (must already be scaffolded). |
| `--source` | no | `` | `` | Raw DDL source directory.  If omitted, the harvest stage is skipped and the existing payload is used.  Mutually exclusive with --source-github. |
| `--source-github` | no | `` | `` | Fetch DDL source from a GitHub repository (e.g. 'myorg/myrepo'). Downloads the repository tarball for --source-ref via the GitHub REST API — no local git clone required.  Mutually exclusive with --source. |
| `--source-ref` | no | `main` | `` | Branch, tag, or commit SHA to fetch when using --source-github (default: main). |
| `--github-token` | no | `` | `` | GitHub personal access token for private repositories.  Falls back to the GITHUB_TOKEN environment variable.  Public repositories work without a token. |
| `--token-map` | no | `` | `` | Path to token_map.conf for harvest token substitution. |
| `--auto-tokenise` | no | `False` | `` | Auto-detect and apply token substitutions in one pass (harvest stage). Equivalent to --auto-tokenise on harvest. |
| `--env-prefix` | no | `` | `` | Env prefix for auto-tokenise token derivation. |
| `--skip-generate` | no | `False` | `` | Skip the generate stage (for projects that do not use the SHIPS view-layer generator). |
| `--inspect-config` | no | `` | `` | Path to inspect.conf (passed to the inspect stage). |
| `--env` | no | `` | `` | Target environment (e.g. DEV). Required to run the package stage. |
| `--env-config` | no | `` | `` | Path to the .conf file for token resolution. Required to run the package stage. |
| `--name` | no | `` | `` | Package name. Required to run the package stage. |
| `--output` | no | `` | `` | Output directory for the built package archive. |
| `--format` | no | `zip` | `zip, tar.gz` | Archive format for the package (default: zip). |
| `--author` | no | `` | `` | Author metadata stamped into the package manifest. |
| `--description` | no | `` | `` | Description metadata stamped into the package manifest. |
| `--commit` | no | `` | `` | Source commit hash stamped into the package manifest. |
| `--strict` | no | `False` | `` | Platform mode: abort the pipeline on the first stage that finishes with errors. Without --strict, all stages run and errors are summarised at the end. |
| `--pause` | no | `False` | `` | Pause after each stage and prompt before continuing. Useful for supervised runs where you want to inspect output before proceeding. Suppressed automatically in CI environments (CI, SHIPS_CI, NO_PROMPT env vars) or when stdout is not a TTY. |

### `rollback`

Build a release package from a previous git tag and print the deploy command to restore that version.  The rollback package is a normal SHIPS package — integrity-checked, trust-scored, and deployable via deploy.py.  Recommended: deploy with --on-drift continue to overwrite any out-of-band changes made after the broken deploy.

```text
usage: td_release_packager rollback [-h] --to-tag TAG --env ENV
                                    --env-config ENV_CONFIG [--name NAME]
                                    [--project PROJECT] [--output OUTPUT]
                                    [--format {zip,tar.gz}]
                                    [--on-drift {abort,skip,continue}]
                                    [--author AUTHOR]
                                    [--description DESCRIPTION]
```

| Argument | Required | Default | Choices | Description |
|---|---:|---|---|---|
| `--to-tag` | yes | `` | `` | Git tag to roll back to (e.g. v1.2.3).  Must exist locally; run 'git fetch --tags' first if the tag is only on the remote. |
| `--env` | yes | `` | `` | Target environment (e.g. PRD).  Must match SHIPS_ENV in --env-config. |
| `--env-config` | yes | `` | `` | Path to the environment .conf file.  Use the CURRENT file — token values come from today's environment, not the tag. |
| `--name` | no | `` | `` | Package name (default: project directory name). |
| `--project` | no | `.` | `` | Project directory containing .build_counter and git repo (default: current directory). |
| `--output` | no | `` | `` | Output directory for the rollback package (default: <project>/releases/). |
| `--format` | no | `zip` | `zip, tar.gz` | Archive format (default: zip). |
| `--on-drift` | no | `continue` | `abort, skip, continue` | Action when schema drift is detected during the subsequent deploy (default: continue — rollback overwrites out-of-band changes). |
| `--author` | no | `` | `` | Author metadata for ships.build.json. |
| `--description` | no | `` | `` | Release description (default: 'Rollback to <tag>'). |

## `database_package_deployer` deployment CLI

Idempotent Teradata DDL Deployment with Restartability. Handles tables, join indexes, hash indexes, secondary indexes, triggers, views, macros, procedures, and functions with mandatory pre-flight validation.

### Global arguments

| Argument | Required | Default | Choices | Description |
|---|---:|---|---|---|
| `-v, --verbose` | no | `False` | `` | Enable debug logging. |

### `deploy`

```text
usage: database_package_deployer deploy [-h] [--from-github OWNER/REPO]
                                        [--release-tag TAG] [--asset FILENAME]
                                        [--pattern PATTERN]
                                        [--order-file ORDER_FILE] [--dry-run]
                                        [--continue-on-error] [--env ENV]
                                        [--approval-code CODE]
                                        [--public-key KEY_FILE] [--host HOST]
                                        [--user USER] [--password PASSWORD]
                                        [--logmech LOGMECH]
                                        [--encryptdata ENCRYPTDATA]
                                        [--sslmode SSLMODE]
                                        [package_dir]
```

| Argument | Required | Default | Choices | Description |
|---|---:|---|---|---|

## Agent-context artefact outputs

The package workflow now emits three additional JSON files into every generated package before archiving:

| File | Produced by | Purpose |
|---|---|---|
| `ships.context.json` | `td_release_packager.context_artifacts.write_context_artifacts` from the package builder | Durable workflow context, constraints, source-of-truth pointers, trust state, governance settings, and evidence references. |
| `ships.manifest.json` | same | Compact agent-safe package inventory, dependency contract, governance summary, token-name summary with values redacted, and evidence map. |
| `ships.handoff.json` | same | Next-actor instructions for a human, CI/CD job, MCP tool, or autonomous deployment agent. |

## SHIPS context entrypoint update

Package generation now emits `ships.index.json` as the canonical package entrypoint. The index contains:

- `entrypoints` with each `ships.*.json` file path, description, required flag, and intended audience
- `recommended_read_order`
- `agent_instructions`

The `decisions.json` audit trail has been renamed to `ships.decisions.json`. CLI, MCP, documentation, tests, and prompts should use `ships.decisions.json` as the canonical filename.

MCP package and verify responses now include:

| Field | Description |
|---|---|
| `package_type` | Identifies the package as `teradata-ships`. |
| `context_entrypoint` | Points to `ships.index.json`, either inside an archive or in an extracted package directory. |
| `required_next_reads` | Ordered list of context files a downstream agent should inspect. |
| `agent_instruction` | Machine-readable instruction telling agents to read `ships.index.json` first and follow `recommended_read_order`. |
